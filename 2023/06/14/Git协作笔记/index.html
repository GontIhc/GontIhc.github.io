Git协作笔记

2023/6/14

Git学习笔记

一、版本库(仓库)安装

repository实质上就是一个文件夹，通过在文件夹里的命令git intit，把该文件里变为一个Git仓库，接着该文件夹里会出现一个隐藏的.git文件夹，这是Git仓库的标志，里面包含了Git版本控制的文件。

二、Git本地仓库

git是一个分布式的版本控制的软件，每个用户都有一个本地仓库，一般也会有一个存放最终版本的远程仓库 **(github就是最常用的远程仓库)**。

对于本地仓库：我们修改或增加文件后，使用git add命令，提交文件到本地仓库的stage中，再通过commit把修改提交到master中。至此才算改动了本地仓库的文件。值得一提的是，在IDEA中进行git协作时，Add操作是自动完成的，即修改文件后，会自动添加到stage中，只需执行commit操作即可完成本地仓库的改动，改动会显示在修改树上。(下图是三次commit，紫色的标签表示现在远程仓库的代码目前所处的状态)，

HEAD是git中指向当前版本的指针(即当前IDEA中显示的代码所处的状态)，紫色的标签是远程仓库所处的状态(该远程仓库是main仓库的一个分支，可以通过merge pull requests把远程仓库的该分支合并到main仓库中)。绿色的标签是本地的dev_lyl仓库目前的状态，紫色的标签则是远程的dev_lyl所处的状态，origin就是远程仓库的意思(起源、开端)。(dev_lyl是远程仓库main的一个分支，当开发了一定的代码后，在远程仓库里可以通过merge pull requests来尝试把dev_lyl合并到main中)。

三、工作流

基本的工作流的过程是：写代码/文件——>Add到本地仓库的Stage区——>commit到本地仓库的master——>push到远程仓库的master——>merge pull requests请求到main仓库。

stage：是本地仓库中的一个暂存区，文件可以通过add 命令放入stage

master：是git自动给我们创建的一个本地分支，修改的文件从stage中commit到master，这个master表示的是本地仓库的主分支，其实本地也有dev分支，我们的开发一般是在dev分支上的。

实际的流程是这样的：

  1. 在远程仓库里有：main分支(master分支)、dev分支(多个，每个开发者一个)

  2. 在个人的本地仓库有：自己的dev分支(当然也可以在本地pull其他人的dev，但是最重要的是自己的)、main(master)分支(从远程main中pull的)，这方便我们掌握主分支的状态。

  3. 提交：在本地的dev_lyl开发后(已经commmit)，再push到远程的dev_lyl，最后再远程仓库里进行merge push requests。

三、实际的工作流(以远哲的工作为例)

  • local仓库存放两个分支：master和dev_lyl。前者用来同步remote仓库的master分支(这是存放项目文件的最核心的分支)，后者用来进行具体的需求代码开发。

  • 在工作区完成代码后，使用commit提交修改的代码到dev_lyl分支上(修改的代码会被idea自动保存在stage区内)，接着要把dev_lyl同步到master分支上，最后把local的master分支合并到remote的master上。

  • rebase(变基)和merge(合并)：

    图中的两句话最终的效果是一样的，merge很好理解，就是把dev_lyl合并到master中，操作的对象是dev_lyl。rebase的意思是在msaster分内中把dev_lyl做过的操作再做一遍，操作的对象是 master。

  • 最后，通过push把local的master合并到remote的master上。
